心得

前言

在這次實作中，我接觸並操作一個以 Flask 為後端、Brython 為前端的專案。一開始面對指令、資料夾結構與啟動流程時感到相當陌生，但透過逐步排錯與實際操作，我逐漸理解了整個系統的架構，以及各元件之間如何協同運作。以下是我對本專案架構與操作流程的理解與心得整理。

一、整體架構理解

本專案屬於典型的 Web 應用程式，採用前後端分離但同一專案管理的方式：

  • 後端（Flask）：負責處理 HTTP 請求、路由控制、資料庫操作，以及將頁面回傳給瀏覽器。

  • 前端（HTML + Brython）：由 Flask 的模板系統載入 HTML，再透過 Brython 讓 Python 程式碼在瀏覽器中執行，負責畫面互動與邏輯呈現。

這樣的架構讓我第一次體會到「Python 不只是在寫後端，也能進到瀏覽器端執行」。

二、資料夾結構的角色分工

在專案目錄中，可以清楚看到 Flask 專案的標準結構：

  • app.py：整個系統的入口。負責建立 Flask 應用、定義路由（例如 /）、啟動伺服器。

  • templates/：存放 HTML 樣板。Flask 透過 render_template() 將這些頁面回傳給使用者。

  • static/：存放前端靜態資源，例如 JavaScript、Brython 程式、CSS、圖片等。

  • robot.sqlite：資料庫檔案，用來保存系統運作時所需的資料。

  • 其他 .py 檔案：輔助功能模組，例如資料建立、資料處理或舊版本備份。

透過實際用 dir 指令查看資料夾，我才真正理解「為什麼一定要在有 app.py 的資料夾執行 python app.py」。

三、啟動流程的實際體會

整個系統的操作流程可以整理為以下步驟：

  1. 進入正確的專案資料夾（確保裡面有 app.py）。

  2. 在命令列執行 python app.py。

  3. Flask 啟動後，在本機開啟一個開發伺服器（127.0.0.1:5000）。

  4. 使用瀏覽器連線到該網址，取得由 Flask 回傳的 HTML 頁面。

  5. HTML 載入後，Brython 在瀏覽器中執行 Python 程式，完成互動功能。

過程中我也學到：

  • 命令列顯示「Running on http://127.0.0.1:5000」代表**成功啟動，不是錯誤**。

  • 網頁一直 loading，通常不是瀏覽器問題，而是後端路由或前端資源卡住。

四、除錯過程帶來的學習

在實作中，我遇到多次錯誤，例如：

  • 找不到 app.py

  • Clone 到錯誤的資料夾（1、2、3）

  • 瀏覽器一直 loading

這些錯誤讓我學會用「系統化的方式排查」：

  • 先確認目前所在目錄

  • 再確認檔案是否存在

  • 接著檢查 Flask 是否有收到請求（cmd 是否出現 GET /）

  • 最後才檢查程式邏輯本身

這種思考方式比單純「重裝或亂試指令」有效得多。

五、學習心得與反思

透過這次操作，我最大的收穫不是寫了多少程式，而是：

  • 真正理解一個 Web 專案「從指令列到瀏覽器」的完整流程

  • 明白資料夾結構不是擺好看的，而是每一層都有責任

  • 體會到錯誤訊息其實是在「提示方向」，而不是單純阻礙

對身為學生的我來說，這是一個從「照指令操作」到「理解系統運作」的重要轉變。

結語

總結來說，這個 Flask × Brython 專案讓我初步建立了 Web 應用的整體觀念，也訓練了我面對錯誤時的分析能力。未來若再接觸類似專案，我會先看結構、再跑程式、最後才修改內容，而不是一開始就感到不知所措。這次的學習經驗，對我而言是非常扎實且有成就感的一次實作。
